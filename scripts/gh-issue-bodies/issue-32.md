## Objective

Preserve the future direction for release, distribution, and adoption so IMPACT can move from a technical product to a broadly usable product without losing discipline.

## Unified Context

IMPACT now has:
- a working scanner
- a public web shell
- a growing activation path
- partial macOS packaging progress
- a live same-origin stats path
- a still-open public npm install gate

**Current** install and distribution work is still **activation-stage** (npm truth, signing/notarization path, staged packaging). This roadmap card exists to keep the broader **future** adoption and release-maturity surface visible while execution stays focused on activation, dashboard truth, and packaging trust.

It is future-facing, not the active sprint spine.

## Theme / Goal

Preserve the longer-range direction for:
- release channels
- packaging options
- install simplicity
- user onboarding
- adoption surfaces beyond the current early technical-user path

## Why this matters

Without a visible release/adoption roadmap:
- install and packaging work may stay tactical
- onboarding may drift away from the real product path
- adoption improvements may be delayed until complexity has already accumulated
- product experience and release practice may diverge

## What this does not mean yet

This issue does **not** mean:
- GUI work now
- DMG-first strategy now
- replacing activation work with packaging-first work
- claiming consumer readiness before install truth and trust work are real

This card is about **future** release/adoption maturity, not GUI/DMG-first execution in the current tranche.

## Risks of misunderstanding

The main risk is using this roadmap theme as permission to prioritise packaging surfaces ahead of install truth and product-live activation.

If misunderstood, the team may:
- treat a DMG as the main product milestone too early
- over-invest in surface polish before the core loop is live
- blur the distinction between a technical artifact and a trustworthy public release path
